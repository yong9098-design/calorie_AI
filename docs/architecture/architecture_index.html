시스템 아키텍처

칼로리 트래커의 전체 구조를 보여줍니다. 단일 HTML SPA가 Vercel Edge Functions을 통해 외부 서비스와 통신합니다.

  • 클라이언트 계층: 모든 코드가 output/index.html 파일 하나에 포함된 SPA
  • 백엔드 API: 3개의 Vercel Edge Function (config, analyze, food-search)
  • 외부 서비스: Supabase (인증/DB), Gemini API (AI 분석), 공공 식품 DB
  • 로컬 스토리지: API 키, 사용자 설정, 임시 캐시 저장

모듈 의존성 그래프

index.html 내부의 JavaScript 모듈 구조를 보여줍니다. 중앙의 state.js가 전역 상태를 관리합니다.

  • 핵심 모듈: boot.js → init.js → state.js (초기화 순서)
  • 화면 제어: nav.js가 각 화면을 라우팅하고 상태를 업데이트
  • 분석 모듈: analyze.js가 이미지를 Base64로 변환해 API에 전송
  • API 클라이언트: 각 화면이 필요한 API를 직접 호출
  • 유틸리티: 모든 모듈이 공통 함수(TDEE 계산, 날짜 포맷팅 등) 사용

음식 분석 데이터 흐름

사진 촬영부터 DB 저장까지의 전체 과정을 시퀀스 다이어그램으로 표현합니다.

  • 단계 1: 사용자가 카메라로 음식 사진 촬영 또는 갤러리에서 선택
  • 단계 2: 프론트엔드에서 이미지를 Base64로 인코딩해 API에 전송
  • 단계 3: Edge Function이 Gemini API 호출 (Vision Input + 한국어 프롬프트)
  • 단계 4: Gemini가 JSON 형식으로 음식명, 칼로리, 단백질, 탄수, 지방 반환
  • 단계 5: Edge Function이 Supabase에 meal_logs 테이블 INSERT
  • 단계 6: 프론트엔드가 결과를 localStorage에 캐싱하고 사용자에게 표시
  • 단계 7: 사용자가 필요시 값 수정 후 최종 저장

  • 프로필 동기화: 온보딩 데이터를 Supabase profiles 테이블에 저장

AI 에이전트 파이프라인 (프로젝트 관리)

칼로리 트래커를 자동으로 생성하는 3-Agent 하네스 구조입니다. 개발 요청이 들어오면 이 파이프라인을 따라 진행됩니다.

  • ① Planner: PRD 읽기 → SPEC.md 설계서 작성 (기능, UI, 데이터 흐름 정의)
  • ② Generator: SPEC 기반으로 HTML/CSS/JS 코드 구현 → index.html 생성
  • ③ Evaluator: SPEC과 코드 검증 → evaluation_criteria.md 기준으로 채점
  • 평가 결과: 합격 → 완료 / 불합격 → Generator로 피드백 반영 (최대 3회 반복)
  • 완료: 합격 후 CHANGELOG.md에 기능, 기술, 파일 변경사항 기록
